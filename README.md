# Fraud Risk Scoring System – IEEE-CIS Fraud Detection ## Overview This project develops a machine learning–based **fraud risk scoring system** for e-commerce transactions using the IEEE-CIS Fraud Detection dataset. Rather than treating fraud detection as a simple classification task, the model produces a **fraud probability score** for each transaction. These scores can support operational decisions such as: - automatically approving low-risk transactions - sending suspicious transactions to manual review - automatically declining high-risk transactions The project is structured to resemble a **real production fraud modeling workflow**, including disciplined data preparation, feature engineering, model comparison, hyperparameter tuning, and out-of-time evaluation. --- # Business Problem Online payment systems process millions of transactions, while only a small percentage are fraudulent. This creates a challenging trade-off: - **Missed fraud** results in financial loss and chargebacks - **False alarms** create friction for legitimate customers and increase operational cost A useful fraud model must therefore do more than predict fraud. It must **prioritize transactions by risk**, allowing fraud teams to focus their investigation capacity where it has the highest impact. --- # Dataset Source: **IEEE-CIS Fraud Detection Dataset (Kaggle)** | Dataset | Rows | Columns | |-------|------|------| | train_transaction | 590,540 | 394 | | train_identity | 144,233 | 41 | Key properties: - Target variable: isFraud - Fraud prevalence: **~3.5%** - Tables joined using TransactionID - Identity information exists for **~25% of transactions** The dataset contains transaction attributes, device information, anonymized behavioral variables, and historical transaction signals. --- # Data Splitting Strategy Fraud patterns change over time, so preventing **temporal leakage** is essential. A time-based split using TransactionDT was applied: | Dataset | Rows | Purpose | |------|------|------| | Development dataset | 472,432 | Model training and tuning | | Holdout dataset | ~118,000 | Final out-of-time evaluation | The holdout dataset represents **future transactions** and was not used during model development. --- # Feature Engineering Feature engineering focused on capturing behavioral signals commonly used in fraud detection. ### Transaction Amount Features - TransactionAmt_log1p - TransactionAmt_is_small - TransactionAmt_is_large These capture nonlinear relationships between transaction size and fraud probability. ### Temporal Features Derived from TransactionDT: - TransactionHour - TransactionDay - TransactionWeek - TransactionHour_sin - TransactionHour_cos Cyclic encoding enables the model to learn daily activity patterns. ### Email Domain Features - P_email_provider - R_email_provider - P_email_suffix - R_email_suffix - email_domain_match - email_suffix_match - both_email_missing - both_email_present These capture consistency patterns in user identity signals. ### Missing Value Indicators Binary indicators were created for selected variables where **missingness itself may carry predictive information**. --- # Modeling Approach The modeling workflow followed a structured pipeline:

Validation performance: | Metric | Value | |------|------| | ROC-AUC | ~0.86 | | PR-AUC | ~0.47 | This established a reference point before evaluating more complex models. --- # Tree-Based Models Three algorithms were compared: - Random Forest - XGBoost - LightGBM Boosting models performed best due to their ability to capture nonlinear relationships and complex feature interactions. --- # Hyperparameter Tuning The LightGBM model was tuned using a focused parameter search over: - number of trees - learning rate - tree depth and leaf size - row and feature sampling - regularization parameters Best validation performance: | Metric | Value | |------|------| | ROC-AUC | ~0.92 | | PR-AUC | ~0.61 | PR-AUC was emphasized because it is more informative for **highly imbalanced fraud datasets**. --- # Final Model The final model is a **tuned LightGBM classifier** trained on the full development dataset. Key parameters:
n_estimators = 500
learning_rate = 0.07
num_leaves = 127
subsample = 0.70
colsample_bytree = 0.80
The model outputs **fraud probability scores** rather than binary predictions. --- # Holdout Evaluation The final model was evaluated on the **out-of-time holdout dataset**. Performance remained strong, confirming that the model generalizes to future transactions. --- # Fraud Risk Policy Simulation Instead of a single classification threshold, the model is used as a **risk scoring engine**. Example operational policy: | Fraud Score | Decision | |------|------| | < 0.20 | Approve | | 0.20 – 0.60 | Manual Review | | ≥ 0.60 | Decline | Observed fraud concentration: | Decision | Fraud Rate | |------|------| | Approve | ~1% | | Manual Review | ~10% | | Decline | ~57% | This separation allows fraud teams to prioritize high-risk transactions while minimizing disruption for legitimate customers. --- # Key Outcomes This project demonstrates a complete machine learning workflow for fraud detection: - structured dataset auditing - careful handling of temporal leakage - domain-informed feature engineering - model benchmarking and tuning - out-of-time validation - translation of model outputs into operational fraud decisions The final model functions as a **fraud risk scoring system capable of supporting transaction monitoring and investigation prioritization**. --- # Tools Used - Python - Pandas - NumPy - Scikit-Learn - LightGBM - Matplotlib - Seaborn
